GitHub MCP Server

A Model Context Protocol server that lets AI tools interact with GitHub — create branches, files, pull requests, and project tasks through a clean tool interface.

Python 3.12+ FastMCP 0.4+ SSE Transport GitHub API v2022-11-28
01

Overview

This server acts as a bridge between any MCP-compatible AI tool (like Claude) and the GitHub API. Instead of writing HTTP requests by hand, you call a named tool — and the server handles authentication, encoding, and error handling for you.

📂

list_files

Browse any directory inside a repository. Supports branches, tags, and commit SHAs.

🌿

create_branch

Create a new branch from any existing branch. Resolves the source SHA automatically.

📄

create_file

Create or update a file with a commit message. Handles Base64 encoding and SHA conflicts.

🔀

create_pull_request

Open a pull request between any two branches, with optional draft mode.

📋

create_project_task

Create a real GitHub Issue and link it to a Projects v2 board — or create a draft card directly on the board. Uses the GraphQL API.

🏗️
Modular Architecture This server now uses a modular package structure! Each tool lives in its own file under github_mcp/tools/, making it easy to maintain, test, and extend. The main server.py is now just 48 lines. See the Helper Functions section for details on the new structure.

Quick Start

Install dependencies

Requires Python 3.12+ and uv (or pip).

bash 
# Clone the repository
git clone <your-repo-url>
cd GitHubMCP

# Install uv (if not already installed)
curl -LsSf https://astral.sh/uv/install.sh | sh

# Create virtual environment and install dependencies
uv venv
source .venv/bin/activate  # On Windows: .venv\Scripts\activate
uv pip install -e .
Create your .env file

Place this in the same directory as server.py.

.env 
GITHUB_TOKEN=ghp_your_token_here
GITHUB_OWNER=your-username
GITHUB_REPO=your-repo
PROJECT_ID=2
Run the server

Starts on port 8090 by default.

bash 
uv run python server.py --port 8090

# Expected output:
# INFO | GitHub MCP server starting at http://0.0.0.0:8090/sse
# INFO | Token: loaded
# INFO | Owner: your-username
# INFO | Repo: your-repo
# INFO | Project ID: 2
Connect MCP Inspector

Open Inspector, set transport to SSE, URL to http://localhost:8090/sse, and click Connect.

Environment Setup

All configuration is loaded from a .env file using python-dotenv. This keeps secrets out of your code.

Variable Required Description
GITHUB_TOKEN yes Personal Access Token (PAT). Needs repo and project scopes. Generate at: GitHub → Settings → Developer settings → Fine-grained tokens.
GITHUB_OWNER default GitHub username or org name. Used as the default owner in all tool calls when not explicitly provided.
GITHUB_REPO default Repository name. Used as the default repo in all tool calls. Can be overridden per call.
PROJECT_ID default GitHub Projects v2 number (shown in the project URL). Default for create_project_task.
🔒
Never commit your .env file Add .env to your .gitignore. If you accidentally push a token to GitHub, revoke it immediately at github.com/settings/tokens.
04

How It Works

Every tool call follows the same path from the AI client through to GitHub and back.

Client MCP Inspector or Claude sends a tool call: {"name": "list_files", "arguments": {}}
↓
FastMCP Receives the message over SSE/stdio, validates arguments, calls the matching Python function
↓
Tool fn Builds the request URL, attaches auth headers via _headers(), sends HTTP request via httpx
↓
GitHub API Processes the request and returns JSON — list of files, branch data, PR info, etc.
↓
Response Tool function parses the JSON, builds a clean result dict, returns it to FastMCP → back to the client

Helper Functions & Modular Structure

The server code is now organized into a modular package structure for better maintainability:

structure 
github_mcp/
├── config.py              # Environment variables & configuration
├── constants.py           # API URLs & GraphQL queries
├── core/
│   └── github_api.py      # HTTP helpers (_headers, _gql_check)
├── utils/
│   └── project_helpers.py # Projects v2 helpers
└── tools/                 # Each tool in its own file
    ├── files.py
    ├── branches.py
    ├── pull_requests.py
    └── ...

_headers() — Authentication (github_mcp/core/github_api.py)

Returns a dict of HTTP headers that every REST API call needs. Raises immediately if the token is missing.

python 
def _headers() -> dict:
    return {
        "Authorization":        f"Bearer {GITHUB_TOKEN}",
        "Accept":               "application/vnd.github+json",
        "X-GitHub-Api-Version": "2022-11-28",
    }
HeaderValueWhy it's needed
Authorization Bearer <token> Proves your identity to GitHub. Without this, every request returns 401 Unauthorized.
Accept application/vnd.github+json Tells GitHub to respond in its enhanced JSON format. Omitting this returns a legacy format with fewer fields.
X-GitHub-Api-Version 2022-11-28 Pins the API version so behaviour doesn't change if GitHub releases a new version.

_gql_headers() — GraphQL Authentication

Extends _headers() with Content-Type: application/json, required when sending a GraphQL query body. Used by project management tools.

_raise_for_status() — Error Guard

Checks the HTTP response status. Any code ≥ 400 triggers a logger.error() log and raises a RuntimeError, stopping the tool immediately.

💡
Why raise instead of returning an error dict? Raising stops execution immediately. If we returned an error dict, the code would continue and try to read fields from a broken response — causing confusing secondary errors far from the real problem.
📦
Benefits of the modular structure:
  • Each tool is ~50-150 lines (easy to read & test)
  • Changes to one tool don't affect others
  • Core utilities are reusable across all tools
  • Main server.py is just 48 lines

Tool: list_files

Lists files and directories at a given path in the repository. Calls the GitHub Contents API.

🔗
GitHub endpoint used GET https://api.github.com/repos/{owner}/{repo}/contents/{path}?ref={branch}
ParameterDefaultDescription
path""Path inside the repo. Empty string = repo root. E.g. "src" or "docs/images".
refNoneBranch name, tag, or commit SHA to read from. None uses the repo's default branch.
ownerenvRepo owner. Falls back to GITHUB_OWNER from .env.
repoenvRepository name. Falls back to GITHUB_REPO from .env.
🤔
Why do we normalise to a list? GitHub's API returns a JSON array when you fetch a directory, but a single JSON object when you fetch a specific file. We wrap single objects in an array so every caller always gets a list — no special-casing needed.
07

Tool: create_branch

Creates a new git branch from an existing branch. Involves two API calls: first to resolve the source branch to a SHA, then to create the new ref.

🔗
GitHub endpoints used GET /repos/{owner}/{repo}/git/ref/heads/{source} → POST /repos/{owner}/{repo}/git/refs
ParameterDefaultDescription
branchrequiredName for the new branch. Convention: feature/thing, fix/bug-name, chore/task.
source_branch"main"Existing branch to copy from. The new branch starts at the same commit.
ownerenvRepo owner.
repoenvRepository name.
💡
Why resolve the SHA first? GitHub's ref creation API requires the exact 40-character commit SHA, not a branch name. We resolve the branch name to its current SHA in step 1, then use that SHA in step 2.
08

Tool: create_file

Creates a new file or updates an existing one, automatically committing the change. Handles Base64 encoding and SHA collision detection.

🔗
GitHub endpoints used GET /repos/{owner}/{repo}/contents/{path} (check existence) → PUT same URL (create/update)
ParameterDefaultDescription
file_pathrequiredPath inside the repo. Include subdirectories: "src/utils/helper.py". Non-existent directories are created automatically.
contentrequiredPlain text content of the file. The tool Base64-encodes it before sending.
commit_messagerequiredGit commit message. Appears in the repo history permanently. Use imperative mood: "Add login helper".
branch"main"Target branch. Should match the branch created in create_branch.
ownerenvRepo owner.
repoenvRepository name.
🚨
Why does updating a file require the existing SHA? GitHub enforces optimistic concurrency. If two people update the same file simultaneously, the second writer must provide the SHA of the version they read. If the file changed in the meantime, the SHA won't match and GitHub rejects the update — preventing silent overwrites.

Tool: create_pull_request

Opens a pull request proposing to merge one branch into another. The PR becomes visible on github.com for code review.

🔗
GitHub endpoint used POST https://api.github.com/repos/{owner}/{repo}/pulls
ParameterDefaultDescription
titlerequiredPR title shown in the PR list. Should summarise the change clearly.
headrequiredSource branch — the one containing your changes. This is the branch you created.
base"main"Target branch — the one you want to merge INTO. Almost always "main" or "develop".
body""PR description. Supports GitHub Flavoured Markdown. Describe what changed and why.
draftFalseIf True, opens a Draft PR — signals work-in-progress, cannot be merged until converted to ready.
ownerenvRepo owner.
repoenvRepository name.

Tool: create_project_task

Creates a task on a GitHub Projects v2 board. Optionally creates a real repository Issue and links it to the board. Uses the GraphQL API (not REST).

🔮
Why GraphQL instead of REST? GitHub Projects v2 is only accessible via GraphQL. The older Projects v1 used REST, but v2 requires GraphQL mutations and queries. This is GitHub's choice, not ours.

Two modes of operation

With repo Creates a real Issue in the repository AND adds it to the project board. Issue has a number, comments, assignees, and labels.
↕
Without repo Creates a Draft Issue — a card that exists ONLY on the project board. Lighter weight, no repository attachment.
ParameterDefaultDescription
titlerequiredTask title shown on the board card and in the issue list.
body""Longer description. Supports Markdown. Shown in the issue/card detail view.
project_numberenvThe number in your project URL: github.com/users/you/projects/2. Defaults to PROJECT_ID.
repoenvIf provided, creates a real Issue in this repo and links it to the project.
ownerenvOwner of both the project and the repo.
assigneeNoneGitHub username to assign. The user receives a notification. Requires repo to be set.
labelNoneLabel name to apply. Must already exist in the repository. E.g. "bug", "enhancement".
🔑
Required token scope Your GITHUB_TOKEN must have the project scope enabled. Without it, the GraphQL query returns null for the project and the tool raises a RuntimeError.

Transport Protocols

MCP supports multiple transport mechanisms for carrying messages between the client (AI tool or Inspector) and the server. Our server uses SSE by default, but FastMCP supports all three.

SSE ✓ Used here

Server-Sent Events — a one-directional HTTP streaming connection. The client connects once and the server can push messages continuously without the client re-requesting.

ProtocolHTTP/1.1 text/event-stream
DirectionServer → Client (push)
PortConfigurable (default 8090)
URL formathttp://host:port/sse
Run commandmcp.run(transport="sse")
Works over a network (remote servers)
Easy to test with MCP Inspector
Persistent connection — low overhead
Firewall/proxy friendly (just HTTP)
Requires a running HTTP server
One-way push only (client uses POST for requests)
stdio Alternative

Standard Input/Output — the simplest transport. Messages are written to stdout and read from stdin. The client process launches the server as a subprocess.

ProtocolJSON lines over stdin/stdout
DirectionBidirectional via pipes
PortNone (no network)
Run commandmcp.run(transport="stdio")
Best forClaude Desktop, local tools
Zero network setup — just run the script
Used by Claude Desktop config
Most secure — no open ports
Lowest latency (in-process pipes)
Only works locally (same machine)
Cannot be tested with MCP Inspector directly
Client must manage the server process lifecycle
Streamable HTTP New in MCP 2025

HTTP + streaming responses — a newer transport that combines regular HTTP POST requests with optional streaming responses. Replaces SSE in newer MCP spec.

ProtocolHTTP POST with chunked transfer
DirectionBidirectional over HTTP
PortConfigurable
Run commandmcp.run(transport="http")
Best forProduction deployments
Full bidirectional communication
Stateless — works with load balancers
Compatible with standard HTTP infrastructure
Supports resumable sessions
More complex setup than SSE
Newer spec — less tool support currently

Switching Transport

Change the last line of server.py to switch transport modes:

python 
# SSE — current mode, for MCP Inspector & remote use
mcp.run(transport="sse", host="0.0.0.0", port=8090)

# stdio — for Claude Desktop config (no host/port needed)
mcp.run(transport="stdio")

# Streamable HTTP — for production / load balanced environments
mcp.run(transport="http", host="0.0.0.0", port=8090)

Claude Desktop Config (stdio)

To use this server with Claude Desktop, use stdio transport and add this to claude_desktop_config.json:

json

        "GITHUB_REPO":   "your-repo",
        "PROJECT_ID":    "2"
      }
    }
  }
}

GraphQL vs REST API

Our server uses two different API styles to talk to GitHub. Four tools use the REST API; one tool — create_project_task — uses the GraphQL API. Understanding the difference explains a lot about how the code is structured.

REST API — The Simple Menu

REST (Representational State Transfer) is the classic way to talk to web APIs. Every resource has a fixed URL and you use HTTP verbs to act on it.

REST — how our tools call GitHub 
# list_files — GET a resource
GET  https://api.github.com/repos/your-username/your-repo/contents/

# create_branch — POST creates a new ref
POST https://api.github.com/repos/your-username/your-repo/git/refs

# create_file — PUT writes file content
PUT  https://api.github.com/repos/your-username/your-repo/contents/hello.md

# create_pull_request — POST opens a PR
POST https://api.github.com/repos/your-username/your-repo/pulls
🍽️
REST is like ordering from a fixed menu Each URL is a dish. You ask for exactly that dish and get everything that comes with it — whether you wanted all of it or not. GET /repos/owner/repo gives you 50+ fields even if you only needed the repo name.

GraphQL — The Custom Order

GraphQL (Graph Query Language) lets you describe exactly what data you want in a single request. Instead of fixed URLs, everything goes to one endpoint: https://api.github.com/graphql. You send a query or mutation in the request body.

GraphQL — how create_project_task works 
# Step 1: Query — find the project's internal node ID
POST https://api.github.com/graphql

{
  "query": "query {
    user(login: \"your-username\") {
      projectV2(number: 2) {
        id        \"← we ONLY ask for these two fields\"
        title
      }
    }
  }"
}

# Response — GitHub returns ONLY what we asked for:
{
  "data": {
    "user": {
      "projectV2": {
        "id":    "PVT_kwDOBqfXXs4A...",
        "title": "My Project Board"
      }
    }
  }
}

# Step 2: Mutation — add an issue to the project
{
  "query": "mutation {
    addProjectV2ItemById(input: {
      projectId: \"PVT_kwDOBqfXXs4A...\"
      contentId: \"I_kwDOBqfXXs5abc123\"
    }) {
      item { id }
    }
  }"
}
🎨
GraphQL is like a custom order at a restaurant Instead of "give me the burger meal", you say "I want the patty, the top bun, and the pickles — no fries, no drink, no bottom bun". You get back exactly and only what you described.

Queries vs Mutations

GraphQL has two types of operations — the equivalent of GET and POST/PUT in REST:

OperationREST equivalentWhat it doesUsed in our code
query GET Read-only. Fetches data without changing anything. Safe to call multiple times. Finding the project node ID from the project number
mutation POST / PUT / DELETE Changes data. Creates, updates, or deletes something. Should only be called once. addProjectV2ItemById (link issue to board) and addProjectV2DraftIssue (create draft card)

Why GitHub Projects v2 Uses GraphQL

GitHub Projects v2 was redesigned from scratch. The old Projects v1 used REST, but v2 is a fully relational data model — projects can contain items from multiple repos, with custom fields and views. Expressing that complexity over REST would require dozens of endpoints. GraphQL handles it with a single flexible query language.

🔗
The "node ID" problem GraphQL works with node IDs — opaque internal identifiers like PVT_kwDOBqfXXs4A.... You can't just pass project_number: 2 to a mutation; you must first resolve the human-readable number (2) into the internal node ID via a query, then use that ID in the mutation. That's the two-step dance in create_project_task.

REST vs GraphQL — Side by Side

Feature REST GraphQL
Endpoint per resource✓ Many URLs/graphql only
Choose exactly which fields to return✗ Gets everything✓ Precise selection
Fetch nested data in one request✗ Multiple calls✓ One query
Human-readable URLs✓✗ POST body only
HTTP caching friendly✓ GET is cacheable✗ All POST
Learning curveLowModerate
Used for GitHub Projects v2✗✓
Used for files, branches, PRs✓✗

The Two-Step Flow in create_project_task

Input You provide project_number=2 (human-readable) and owner="your-username"
↓
Query GraphQL query resolves 2 → internal node ID "PVT_kwDO..." (try user first, then organization)
↓
REST call (If repo given) POST to /repos/.../issues — creates a real Issue, returns its node ID
↓
Mutation A addProjectV2ItemById — links the Issue node ID to the project node ID
Mutation B (If no repo) addProjectV2DraftIssue — creates a draft card directly on the board
↓
Result Returns item_id, project_title, project_id, and (if real issue) issue_url

Error Handling

Common GitHub API errors and what they mean:

StatusMeaningFix
401 Unauthorized Token is invalid, expired, or missing. Check GITHUB_TOKEN in .env.
403 Forbidden Token doesn't have the required scope. For Projects: add project scope. For repos: add repo scope.
404 Not Found Repo, branch, file, or owner doesn't exist. Check spelling and that the repo is accessible with your token.
422 Unprocessable Often means: branch already exists (create_branch), or PR already exists between those branches.
409 Conflict SHA mismatch on file update. The file changed since you last read it. Re-call to get the fresh SHA and retry.

Full Workflow Example

A complete sequence using all 5 tools in order:

1. Explore list_files(path="") → See what's in the repo root
↓
2. Branch create_branch(branch="feature/add-readme") → Safe copy from main
↓
3. Write create_file(file_path="README.md", content="# Hello", commit_message="Add README", branch="feature/add-readme")
↓
4. Review create_pull_request(title="Add README", head="feature/add-readme") → Open for review
↓
5. Track create_project_task(title="Review README PR", assignee="your-username") → Pin to project board
