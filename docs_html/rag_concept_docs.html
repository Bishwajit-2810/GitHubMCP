TASK 3.1

Better Chunking Strategies

How you split text dramatically affects what gets retrieved — and whether retrieved chunks are useful.

⚠️
The Problem: A fixed 500-character split cuts mid-sentence, mid-function, or mid-idea. The LLM then gets an incomplete chunk that may be useless or misleading.
CHUNKING STRATEGY COMPARISON

① RecursiveCharacter

def hello():
print("hi")
← CUT HERE (500 chars)
return True
class Foo:

⚠ Can split mid-function

② PythonCode

def hello():
print("hi")
return True
class Foo:
pass

✅ Splits at class/function

③ MarkdownHeader

## Installation
Run pip install...
## Usage
Import the module...

✅ Splits at ## / ### headers

① RecursiveCharacterTextSplitter Baseline

The default splitter. Tries a list of separators in order (\n\n, \n, , "") until the chunk is small enough. Best for general text. 

from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,        # max characters per chunk
    chunk_overlap=50,      # chars shared between adjacent chunks
    length_function=len,   # how to measure length (can use token counter)
    separators=[            # tried left-to-right until chunk fits
        "\n\n",            # paragraph break (preferred)
        "\n",             # line break
        " ",              # word break
        ""               # character-by-character (last resort)
    ]
)

docs = splitter.split_documents(raw_docs)
print(f"Created {len(docs)} chunks")
Parameter Type Effect
chunk_size int Maximum characters (or tokens) per chunk. Smaller = more precise retrieval but more chunks.
chunk_overlap int Characters shared between consecutive chunks. Prevents context loss at boundaries. ~10% of chunk_size is typical.
separators list[str] Tried in order. If splitting by \n\n keeps chunk under limit, that's used — otherwise falls to next.
length_function callable How to count length. Use tiktoken counter for token-aware splitting.

② PythonCodeTextSplitter For .py files

Pre-configured to split at Python-aware boundaries: class definitions, function definitions, decorators, and then falls back to general separators. 

from langchain.text_splitter import Language, RecursiveCharacterTextSplitter

# Option A: explicit Python splitter
py_splitter = RecursiveCharacterTextSplitter.from_language(
    language=Language.PYTHON,
    chunk_size=1000,   # larger for code — functions can be long
    chunk_overlap=100
)

# Under the hood, separators become Python-aware:
# ["\nclass ", "\ndef ", "\n\tdef ", "\n\n", "\n", " ", ""]

py_docs = py_splitter.split_documents(python_files)

# Example: shows metadata preserved
for doc in py_docs[:3]:
    print(doc.metadata)  # {'source': 'app.py', 'language': 'python'}
    print(doc.page_content[:80])
✅ Best For: A Python codebase where users ask "how does the authenticate function work?" — each function is its own chunk, so the right chunk gets retrieved.
⚠ Avoid When: Splitting Markdown or plain text — it will still split at \ndef which appears nowhere, so it degrades to character splitting anyway.

③ MarkdownHeaderTextSplitter For .md files

Splits at Markdown headers (#, ##, ###) and includes the header hierarchy as metadata. Extremely useful for docs/wikis. 

from langchain.text_splitter import MarkdownHeaderTextSplitter

# Define which headers to split on + what metadata key to use
headers_to_split = [
    ("#",   "h1"),   # top-level section
    ("##",  "h2"),   # subsection
    ("###", "h3"),   # sub-subsection
]

md_splitter = MarkdownHeaderTextSplitter(
    headers_to_split_on=headers_to_split,
    strip_headers=False    # keep the header text inside the chunk
)

md_docs = md_splitter.split_text(markdown_string)

# Each chunk carries rich metadata:
# {'h1': 'Installation', 'h2': 'Linux Setup', 'source': 'README.md'}

# ── Chain with size-based splitter for very long sections ──
from langchain.text_splitter import RecursiveCharacterTextSplitter

size_splitter = RecursiveCharacterTextSplitter(chunk_size=500, chunk_overlap=30)
final_docs = size_splitter.split_documents(md_docs)
Parameter Type Effect
headers_to_split_on list[tuple] Each tuple is (marker, metadata_key). The header text becomes that key's value in chunk metadata.
strip_headers bool If True, removes the ## line from the chunk body (it's still in metadata). False keeps it for context.

TASK 3.2

Re-Ranking Retrieved Results

Vector similarity finds candidates — re-ranking finds the truly useful ones.

⚠️
The Problem: Top-K by cosine similarity retrieves what is semantically close to the query embedding — but that's not always what's most useful to the LLM. A chunk that mentions "authentication" many times scores high even if it doesn't answer the question.
RE-RANKING PIPELINE (ContextualCompressionRetriever)
1
User Query
"How does token refresh work in the auth module?"
2
Base Retriever (ChromaDB)
Embeds query → cosine similarity → returns top-20 candidate chunks
3
FlashrankRerank (Cross-Encoder)
Scores each (query, chunk) pair together — sees both simultaneously — outputs relevance score
4
Top-5 Re-ranked Chunks
Only the most relevant chunks are passed to the LLM — reducing noise and hallucination
5
LLM generates answer
With high-quality context, grounded and accurate response

🔢 Bi-Encoder (used in retrieval)

  • Encodes query and chunk separately
  • Fast — precompute chunk embeddings
  • Score = cosine(emb_q, emb_chunk)
  • Doesn't see full context of both together
  • Good for initial candidate fetch (top-20)
VS

🎯 Cross-Encoder (used in re-ranking)

  • Sees query + chunk together
  • Slower — can't precompute
  • Much more accurate relevance score
  • Run only on top-20 candidates (feasible)
  • Flashrank runs locally — free & fast

💻 Implementation — ContextualCompressionRetriever + FlashrankRerank 

from langchain.retrievers import ContextualCompressionRetriever
from langchain.retrievers.document_compressors import FlashrankRerank
from langchain_chroma import Chroma

# ── Step 1: base retriever fetches wide candidates ──
base_retriever = vectorstore.as_retriever(
    search_type="similarity",
    search_kwargs={"k": 20}  # fetch 20 — more = better recall
)

# ── Step 2: Flashrank cross-encoder re-ranker ──
compressor = FlashrankRerank(
    model="ms-marco-MiniLM-L-12-v2", # default, ~33MB, runs locally
    top_n=5                            # keep only top 5 after reranking
)

# ── Step 3: Combine into compression retriever ──
reranking_retriever = ContextualCompressionRetriever(
    base_compressor=compressor,
    base_retriever=base_retriever
)

# ── Step 4: Drop into your existing LCEL chain ──
from langchain_core.runnables import RunnablePassthrough
from langchain_core.output_parsers import StrOutputParser

rag_chain = (
    {"context": reranking_retriever, "question": RunnablePassthrough()}
    | prompt
    | llm
    | StrOutputParser()
)
Parameter Type Effect
model str Flashrank model name. ms-marco-MiniLM-L-12-v2 balances speed vs quality. ms-marco-TinyBERT-L-2-v2 is faster/smaller.
top_n int How many chunks survive re-ranking. 3–5 is typical. More = more context but higher token cost.
base retriever k int Initial candidates. Set to 20–50 for good recall. Re-ranker handles narrowing down.

TASK 3.3

HyDE — Hypothetical Document Embeddings

Short questions embed very differently from long code chunks. HyDE bridges that gap.

⚠️
The Problem: "How do I reset my password?" (8 words) embeds to a very different vector than a 500-char code block explaining the reset flow — even though the code block is exactly what you need. The embedding spaces don't align well.
HyDE vs NAIVE RAG RETRIEVAL
❌ NAIVE RAG
❓ Short Question"How does auth work?"
↓ embed directly
🔢 Short Query Vectorsmall, sparse signal
↓ similarity search
😕 Mismatchcode chunks are long & dense
✅ HyDE
❓ Short Question"How does auth work?"
↓ LLM generates hypothetical answer
📝 Hypothetical Doc"The auth module uses JWT tokens. The authenticate() function..."
↓ embed hypothetical doc
✅ Rich Vectorsame space as real code chunks

💻 HypotheticalDocumentEmbedder — Full Implementation

from langchain.chains import HypotheticalDocumentEmbedder
from langchain_openai import ChatOpenAI, OpenAIEmbeddings
from langchain_core.prompts import ChatPromptTemplate
from langchain_chroma import Chroma

# ── The prompt that generates a hypothetical answer ──
hyde_prompt = ChatPromptTemplate.from_template("""
You are an expert assistant. Given the question below, write a 
hypothetical passage from a technical document that would answer it.
Write it as if extracted from real documentation — detailed and specific.

Question: {QUESTION}
Passage:""")

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)
base_embeddings = OpenAIEmbeddings()

# ── Wrap embeddings with HyDE ──
hyde_embeddings = HypotheticalDocumentEmbedder.from_llm(
    llm=llm,
    base_embeddings=base_embeddings,
    prompt=hyde_prompt,
    n=1   # generate 1 hypothetical doc; use n>1 and average for robustness
)

# ── Use these embeddings as the retriever's query encoder ──
vectorstore = Chroma(
    embedding_function=base_embeddings,  # docs embedded normally
    persist_directory="./chroma_db"
)

hyde_retriever = vectorstore.as_retriever(
    search_kwargs={"k": 5}
)

# ── Inject HyDE into the chain ──
from langchain_core.runnables import RunnableLambda

def hyde_retrieve(question: str):
    # 1. generate hypothetical doc
    hypo_doc = (hyde_prompt | llm).invoke({"QUESTION": question})
    # 2. embed the hypothetical doc, search with that vector
    return vectorstore.similarity_search(hypo_doc.content, k=5)

rag_chain = (
    {"context": RunnableLambda(hyde_retrieve), "question": RunnablePassthrough()}
    | prompt | llm | StrOutputParser()
)
Parameter Type Effect
n int Number of hypothetical docs to generate. Multiple are averaged → more stable embedding but higher latency.
temperature float 0 = deterministic hypothetical doc. Higher = more varied but may drift from ground truth.
prompt ChatPromptTemplate The generation prompt. Tailor it to your domain — "Write a passage from Python docs" vs "Write a medical case note".

TASK 3.4

ChromaDB vs PGVector

Two powerful vector backends — each shines in different deployment contexts.

🟦 ChromaDB

  • File-based (SQLite + HNSW index)
  • Zero external dependencies
  • One line setup: Chroma(...)
  • Great for local dev & notebooks
  • Client/server mode for prod
  • No SQL JOIN capability
  • Inspect via Chroma UI
VS

🟨 PGVector

  • Lives inside PostgreSQL
  • Requires Postgres setup
  • Full ACID guarantees
  • JOIN vectors with relational data
  • Use if app already uses Postgres
  • Scale with existing Postgres infra
  • Inspect via psql / pgAdmin

💻 Swapping Backends — Identical Chain Code

LangChain's VectorStore abstraction means only one line changes when you swap backends: 

from langchain_chroma import Chroma
from langchain_postgres import PGVector

# ─── ChromaDB backend ───
chroma_store = Chroma(
    collection_name="my_docs",
    embedding_function=embeddings,
    persist_directory="./chroma_db"  # local folder
)

# ─── PGVector backend ───
pgvector_store = PGVector(
    connection="postgresql+psycopg://user:pass@localhost/mydb",
    collection_name="my_docs",
    embedding_function=embeddings,
    use_jsonb=True     # store metadata as JSONB for fast filtering
)

# ─── THE REST IS IDENTICAL ───
def build_rag_chain(vectorstore):
    retriever = vectorstore.as_retriever(
        search_type="similarity",
        search_kwargs={"k": 5}
    )
    return (
        {"context": retriever, "question": RunnablePassthrough()}
        | prompt | llm | StrOutputParser()
    )

chain_chroma   = build_rag_chain(chroma_store)   # swap here
chain_pgvector = build_rag_chain(pgvector_store) # swap here

# ─── Metadata filtering — same API, both stores ───
chroma_store.as_retriever(search_kwargs={
    "k": 5,
    "filter": {"filetype": "py"}
})

pgvector_store.as_retriever(search_kwargs={
    "k": 5,
    "filter": {"filetype": "py"}
})

TASK 3.5

Metadata Filtering with Self-Query Retriever

Let the LLM automatically construct metadata filters from natural language questions.

⚠️
The Problem: When you ask "How do I configure the database?" — naive RAG retrieves from all files. It may return a Python migration script instead of the README setup guide. You want to automatically filter to filetype: "md" for setup questions and filetype: "py" for code questions.
SELF-QUERY RETRIEVER FLOW
1
Natural Language Question → "How do I install the Python package?"
2
LLM Query Constructor
Analyzes the question and generates: {"filter": {"filetype": "md"}, "query": "install Python package"}
3
Filtered Vector Search
Searches only .md files — Python code files are excluded entirely
4
Precise, relevant chunks returned — from the right file type

💻 SelfQueryRetriever — Full Implementation

from langchain.retrievers.self_query.base import SelfQueryRetriever
from langchain.chains.query_constructor.base import AttributeInfo
from langchain_openai import ChatOpenAI

# ── Step 1: Describe your metadata fields ──
metadata_field_info = [
    AttributeInfo(
        name="source",
        description="The filename the chunk came from, e.g. 'app.py' or 'README.md'",
        type="string"
    ),
    AttributeInfo(
        name="filetype",
        description="File extension: 'py' for Python code, 'md' for documentation",
        type="string",
        # LLM knows valid values from description
    ),
]

# ── Step 2: Describe your document collection ──
document_content_description = """
Technical documentation and Python source code for a web application.
Contains setup guides (md files) and implementation code (py files).
"""

# ── Step 3: Build the self-query retriever ──
self_query_retriever = SelfQueryRetriever.from_llm(
    llm=ChatOpenAI(model="gpt-4o-mini", temperature=0),
    vectorstore=vectorstore,
    document_contents=document_content_description,
    metadata_field_info=metadata_field_info,
    verbose=True,           # prints the generated filter — great for debugging
    enable_limit=True,      # allow "give me 3 results" phrasing
    search_kwargs={"k": 5}
)

# ── Step 4: Drop in as the retriever ──
rag_chain = (
    {"context": self_query_retriever, "question": RunnablePassthrough()}
    | prompt | llm | StrOutputParser()
)
Parameter Type Effect
metadata_field_info list[AttributeInfo] Tells the LLM what metadata exists to filter on. The better your descriptions, the better the auto-generated filters.
document_contents str Describes the overall document collection. Helps LLM understand when to apply filters.
enable_limit bool Allows questions like "give me 2 examples" to set k=2 automatically.
verbose bool Logs the structured query the LLM generates — essential for debugging.

TASK 3.6

RAGAS — Evaluate RAG Quality

Stop guessing. Measure faithfulness, relevancy, and recall with automated metrics.

🔬
RAGAS (Retrieval-Augmented Generation Assessment) gives you three key metrics that together tell you why your pipeline is failing — not just that it is failing.
THREE RAGAS METRICS EXPLAINED

⛪ Faithfulness

Is the answer grounded in the retrieved context?

RAGAS breaks the answer into claims and checks if each is supported by the retrieved chunks. Score = claims_supported / total_claims

Low score means: hallucination — LLM is making things up beyond the context

🎯 Answer Relevancy

Does the answer address the question?

Generates N questions from the answer and checks if they match the original question using cosine similarity of embeddings.

Low score means: the answer is factual but off-topic or too vague

📚 Context Recall

Did retrieval find the right chunks?

Compares retrieved chunks against ground-truth answers. Checks what fraction of the ground-truth content was actually retrieved.

Low score means: retrieval is missing key information — fix chunking or retrieval

💻 Full RAGAS Evaluation Setup

from ragas import evaluate
from ragas.metrics import faithfulness, answer_relevancy, context_recall
from datasets import Dataset

# ── Step 1: Build evaluation dataset ──
# You need: questions, ground_truth answers, and pipeline outputs

questions = [
    "How does token authentication work?",
    "What environment variables are required?",
    # ... 8 more
]

ground_truths = [
    "Token auth uses JWT. The authenticate() function validates the token...",
    "Required env vars are: DATABASE_URL, SECRET_KEY, DEBUG...",
    # ... 8 more
]

# ── Step 2: Run your RAG pipeline on all questions ──
answers = []
contexts = []

for q in questions:
    result = rag_chain.invoke(q)
    answers.append(result["answer"])
    # collect the retrieved chunks for context
    contexts.append([doc.page_content for doc in result["source_documents"]])

# ── Step 3: Build RAGAS Dataset ──
eval_data = {
    "question":  questions,
    "answer":    answers,
    "contexts":  contexts,         # list of lists of strings
    "ground_truth": ground_truths
}

dataset = Dataset.from_dict(eval_data)

# ── Step 4: Evaluate ──
result = evaluate(
    dataset,
    metrics=[
        faithfulness,      # no ground truth needed
        answer_relevancy,  # no ground truth needed
        context_recall,    # needs ground_truth
    ]
)

print(result.to_pandas())
# Output:
#    faithfulness  answer_relevancy  context_recall
# 0          0.82              0.75            0.68
Field Type Notes
question list[str] The test questions
answer list[str] Your RAG pipeline's output for each question
contexts list[list[str]] The chunks retrieved for each question — each question gets a list of chunks
ground_truth list[str] Manually written ideal answer. Required for context_recall and context_precision.
