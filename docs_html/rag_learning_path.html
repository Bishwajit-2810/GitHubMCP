Task 1.1

Understanding the RAG Architecture

What is RAG, why does it exist, and how do its three core components connect? Build the mental model before writing any code.

TASK 1.1

What is RAG — and Why Does It Exist?

The Problem with Pure LLMs

🧠 Hallucination

LLMs generate plausible-sounding text that can be factually wrong. They have no way to say "I don't know" — they just produce something confident.

📅 Staleness

Training data has a cutoff date. The model knows nothing about code written yesterday or docs changed last week.

📏 Context Limits

You can't paste a 100 000-line codebase into a prompt. Even with large windows, it's slow and expensive.

💡
RAG solves all three: Retrieve only the relevant pieces at query time → ground the LLM in real, current documents → no hallucination about your specific codebase.

The 3 Core Components

1

Ingestion — "Loading the library"

Take raw documents, split them into chunks, convert each chunk into a vector (a list of numbers encoding meaning), and store those vectors in a searchable database. Done once, or whenever docs change.

2

Retrieval — "Finding the right pages"

Convert the user's question to a vector using the same embedding model. Find chunks whose vectors are closest in meaning. Return the top-K most relevant.

3

Generation — "Writing the answer"

Pass retrieved chunks + question into a prompt. Send to LLM. The model answers only from provided context — it cannot hallucinate facts not in those chunks.

🎓
3-minute explanation: "RAG is like giving an AI a textbook and a highlighter. Before answering, it finds the most relevant paragraphs in the book, highlights them, and reads only those — instead of guessing from memory."

Cosine similarity ranges from −1 to 1. Score 1 = identical direction (same meaning). Score 0 = unrelated. For RAG we retrieve the chunks with the highest similarity to the query vector.

💻 Embedding Experiment 

from langchain_huggingface import HuggingFaceEmbeddings
import numpy as np

embeddings = HuggingFaceEmbeddings(
    model_name="all-MiniLM-L6-v2",   # 384-dim, ~80 MB, free, fast
    model_kwargs={"device": "cpu"},
    encode_kwargs={"normalize_embeddings": True}  # L2 norm → cosine = dot
)

sentences = [
    "create a branch",
    "make a new git branch",
    "checkout to a feature branch",
    "what's the weather today",
    "how do I authenticate with GitHub",
]

vecs = embeddings.embed_documents(sentences)
query = vecs[0]  # "create a branch"

for sent, vec in zip(sentences[1:], vecs[1:]):
    sim = np.dot(query, vec)   # normalised → dot product = cosine
    print(ff"{sim:.4f}  {sent}")

# 0.9712  "make a new git branch"    ← same intent
# 0.8934  "checkout to a feature"     ← git-related
# 0.1203  "what's the weather"        ← unrelated
# 0.5621  "authenticate with GitHub"  ← GitHub, different action
Parameter Type Explanation
model_name str all-MiniLM-L6-v2 — best balance of speed vs quality for code search. Use all-mpnet-base-v2 for higher quality (2× slower).
device str "cpu" for local dev; "cuda" for GPU. CPU is fine for ingestion of small codebases.
normalize_embeddings bool When True, vectors have length 1. Cosine similarity then equals dot product — faster and numerically stable.
🎯
Key insight: "create a branch" ≈ "make a new git branch" because the model learned from billions of documents that these phrases appear in identical contexts. Word overlap is irrelevant — meaning is what's compared.

Task 2.1

Data Ingestion — ingest.py

Load files, split into chunks, embed, write to both ChromaDB and PGVector simultaneously.

TASK 2.1

GithubFileLoader → Splitter → Embedder → Store

Ingestion Pipeline
1

GithubFileLoader

Fetches .py and .md files from the repo via GitHub API. Each file → a Document with source metadata pre-attached.

2

RecursiveCharacterTextSplitter (chunk_size=500, overlap=50)

Splits long files into ~500-char chunks with 50-char overlap to preserve context at boundaries. Each chunk inherits the parent file's metadata.

3

HuggingFaceEmbeddings — all-MiniLM-L6-v2

Each chunk's text → 384-float vector. Processed in batches.

4

Write to ChromaDB + PGVector

Same chunks stored in both backends. Enables backend comparison in Phase 3 without re-ingesting. 

# ingest.py
from langchain_community.document_loaders import GithubFileLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_huggingface import HuggingFaceEmbeddings
from langchain_chroma import Chroma
from langchain_postgres import PGVector

# 1. Load from GitHub
loader = GithubFileLoader(
    repo="your-org/GitHubMCP",
    access_token="ghp_...",
    file_filter=lambda p: p.endswith((".py", ".md"))
)
raw_docs = loader.load()

# 2. Tag filetype, then split
for doc in raw_docs:
    doc.metadata["filetype"] = doc.metadata["source"].split(".")[-1]

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,     # max chars per chunk
    chunk_overlap=50,   # preserve context at boundaries
)
chunks = splitter.split_documents(raw_docs)

# 3. Embeddings
emb = HuggingFaceEmbeddings(
    model_name="all-MiniLM-L6-v2",
    encode_kwargs={"normalize_embeddings": True}
)

# 4a. ChromaDB
Chroma.from_documents(chunks, emb,
    collection_name="github_mcp",
    persist_directory="./chroma_db")

# 4b. PGVector
PGVector.from_documents(chunks, emb,
    collection_name="github_mcp",
    connection="postgresql+psycopg://postgres:pass@localhost/ragdb",
    pre_delete_collection=True)

print(ff"✅ Indexed {len(chunks)} chunks from {len(raw_docs)} files.")
Parameter Type Explanation
file_filter lambda Called per file path. Return True to include. Excludes binaries, lock files, etc.
chunk_size int 500 chars ≈ 100–120 tokens. Small enough to be specific, large enough to hold a complete thought or function.
chunk_overlap int 50 chars shared between adjacent chunks. Prevents losing context when a sentence spans a boundary.
pre_delete_collection bool PGVector: drops and recreates collection on each run. Use in dev; set False in production.
from_documents() classmethod Embeds all docs and stores them in one call — handles batching internally.
Task 2.2

Build the RAG Chain with LCEL

LCEL composes chain steps with the pipe operator. Each step's output becomes the next step's input.

TASK 2.2

rag_query.py — The LCEL Pipeline

LCEL Data Flow
{ "context": retriever, "question": RunnablePassthrough() } → {context, question}
| format_docs → joins retrieved chunks into a single string RunnableLambda
| prompt → fills template: {context} + {question} ChatPromptTemplate
| llm → ChatAnthropic generates answer ChatAnthropic
| StrOutputParser() → AIMessage → plain str → str 
# rag_query.py
from langchain_chroma import Chroma
from langchain_huggingface import HuggingFaceEmbeddings
from langchain_anthropic import ChatAnthropic
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.runnables import RunnablePassthrough, RunnableLambda
from langchain_core.output_parsers import StrOutputParser

vectorstore = Chroma(collection_name="github_mcp",
    embedding_function=HuggingFaceEmbeddings(model_name="all-MiniLM-L6-v2"),
    persist_directory="./chroma_db")

retriever = vectorstore.as_retriever(
    search_type="similarity",   # or "mmr" for diverse results
    search_kwargs={"k": 5}
)

def format_docs(docs):
    return "\n\n---\n\n".join(
        ff"[File: {d.metadata.get('source','?')}]\n{d.page_content}"
        for d in docs
    )

prompt = ChatPromptTemplate.from_template("""
Use ONLY the context below to answer. If the answer is not in
the context, say "I don't have that information in the indexed files."
Always cite the source filename.

Context:
{context}

Question: {question}
Answer:""")

rag_chain = (
    {"context": retriever | RunnableLambda(format_docs),
     "question": RunnablePassthrough()}
    | prompt
    | ChatAnthropic(model="claude-sonnet-4-5", temperature=0)
    | StrOutputParser()
)

# Test
for q in ["How do I add a new MCP tool?",
           "What environment variables are required?",
           "How does GitHub API authentication work?"]:
    print(rag_chain.invoke(q))
Component Role Key Parameters
as_retriever() Wraps VectorStore as a Runnable search_type: "similarity" or "mmr" (diverse). k: how many chunks.
RunnablePassthrough() Passes the input unchanged Routes original question alongside retrieved context.
temperature LLM randomness 0 = deterministic (best for Q&A). 0.7 = creative.
StrOutputParser() Extracts plain text Converts AIMessage(content="…") → Python str.
"mmr" search_type Max Marginal Relevance Penalises redundant chunks → 5 diverse relevant chunks instead of 5 near-duplicates.
⚠️
Grounding is in the prompt: The LLM only stays grounded because the prompt says "Use ONLY the context." Without this instruction, the model mixes retrieved content with training knowledge.

def split_by_type(docs):

Parameter Type Explanation
chunk_size int Max characters per chunk. Smaller = more precise retrieval but more chunks. For code use 1000 (functions are long).
chunk_overlap int Characters shared between consecutive chunks. ~10% of chunk_size prevents losing context at cuts.
separators list[str] Tried in order. If \n\n keeps chunk under limit it's used; otherwise falls to \n, then space, then character.
headers_to_split_on list[tuple] Each tuple: (marker, metadata_key). Header text becomes that key's value in chunk metadata — enables header-aware filtering.
strip_headers bool False keeps the ## line in chunk body. True removes it (still in metadata). False = more context for LLM.

Task 3.2

Re-Ranking Retrieved Results

Vector similarity finds candidates — a cross-encoder re-ranker finds the truly useful ones.

TASK 3.2

ContextualCompressionRetriever + FlashrankRerank

⚠️
Problem: Top-K by cosine similarity retrieves what is semantically close to the query vector — but that's not always what's most useful. A chunk mentioning "authentication" repeatedly scores high even if it doesn't answer the question.
Re-Ranking Pipeline
1

User Query

"How does token refresh work in the auth module?"

2

Base Retriever (ChromaDB) — k=20

Embeds query → cosine similarity → returns 20 candidate chunks. Wide net for high recall.

3

FlashrankRerank (Cross-Encoder)

Scores each (query, chunk) pair together — sees both simultaneously. Far more accurate relevance signal than embedding similarity.

4

Top-5 Re-ranked Chunks → LLM

Only the most relevant chunks reach the LLM. Reduces noise and hallucination.

🔢 Bi-Encoder (retrieval)

  • Encodes query and chunk separately
  • Fast — precompute chunk embeddings
  • score = cosine(emb_q, emb_chunk)
  • Doesn't see both texts together
  • Good for initial fetch (top-20)
VS

🎯 Cross-Encoder (re-rank)

  • Sees query + chunk together
  • Slower — can't precompute
  • Much more accurate relevance
  • Only runs on top-20 candidates
  • Flashrank: free, runs locally
from langchain.retrievers import ContextualCompressionRetriever
from langchain.retrievers.document_compressors import FlashrankRerank

# Step 1: base retriever — wide net
base_retriever = vectorstore.as_retriever(
    search_type="similarity",
    search_kwargs={"k": 20}   # more candidates = better recall
)

# Step 2: cross-encoder re-ranker
compressor = FlashrankRerank(
    model="ms-marco-MiniLM-L-12-v2",  # ~33 MB, runs locally, free
    top_n=5                             # keep only top 5 after reranking
)

# Step 3: combine
reranking_retriever = ContextualCompressionRetriever(
    base_compressor=compressor,
    base_retriever=base_retriever
)

# Step 4: drop into LCEL chain — one line change
rag_chain = (
    {"context": reranking_retriever | RunnableLambda(format_docs),
     "question": RunnablePassthrough()}
    | prompt | llm | StrOutputParser()
)
Parameter Type Explanation
model str ms-marco-MiniLM-L-12-v2 balances speed vs quality. ms-marco-TinyBERT-L-2-v2 is faster/smaller.
top_n int How many chunks survive re-ranking. 3–5 typical. More = more context but higher token cost.
base retriever k int Set to 20–50 for good recall. The re-ranker handles narrowing down from there.

from langchain_core.prompts import ChatPromptTemplate
from langchain_core.runnables import RunnableLambda

# ── Custom HyDE prompt ──
hyde_prompt = ChatPromptTemplate.from_template("""
You are an expert assistant. Given the question below, write a 
hypothetical passage from technical documentation that would answer it.
Write it as if extracted from real code documentation — detailed and specific.

Question: {question}
Passage:""")

# ── HyDE retrieval function ──
def hyde_retrieve(question: str):
    # 1. generate hypothetical document
    hypo_doc = (hyde_prompt | llm).invoke({"question": question})
    # 2. embed the hypothetical doc and search
    return vectorstore.similarity_search(hypo_doc.content, k=5)

# ── Drop into the LCEL chain ──
rag_chain = (
    {"context": RunnableLambda(hyde_retrieve) | RunnableLambda(format_docs),
     "question": RunnablePassthrough()}
    | prompt | llm | StrOutputParser()
)
Concept Type Explanation
hyde_prompt ChatPromptTemplate Instructs LLM to write a hypothetical passage. Tailor to your domain — "Write a passage from Python docs" vs "Write a medical case note".
temperature float 0 = deterministic hypothetical doc. Higher = varied but may drift from ground truth.
similarity_search(hypo_doc) call Embeds the hypothetical doc (not the query) and uses that richer vector for retrieval.

from langchain.retrievers.self_query.base import SelfQueryRetriever
from langchain.chains.query_constructor.base import AttributeInfo

# ── Describe your metadata fields ──
metadata_field_info = [
    AttributeInfo(
        name="source",
        description="The filename, e.g. 'app.py' or 'README.md'",
        type="string"
    ),
    AttributeInfo(
        name="filetype",
        description="File extension: 'py' for Python code, 'md' for docs",
        type="string"
    ),
]

# ── Describe the document collection ──
doc_description = """
Technical documentation and Python source code for a web application.
Contains setup guides (.md) and implementation code (.py).
"""

# ── Build the self-query retriever ──
self_query_retriever = SelfQueryRetriever.from_llm(
    llm=ChatAnthropic(model="claude-sonnet-4-5", temperature=0),
    vectorstore=vectorstore,
    document_contents=doc_description,

)

# Drop in as retriever — no other chain changes needed

    | prompt | llm | StrOutputParser()
)

Task 3.6 — Bonus / Capstone

Evaluate RAG Quality with RAGAS

Stop guessing — measure faithfulness, answer relevancy, and context recall with automated metrics.

TASK 3.6

Three RAGAS Metrics Explained + Full Setup

⛪ Faithfulness

Is the answer grounded in the context?

Breaks the answer into claims, checks if each is supported by retrieved chunks. Score = claims_supported / total_claims.

Low score means: hallucination — LLM invents facts not in context.

🎯 Answer Relevancy

Does the answer address the question?

Generates N questions from the answer, checks if they match the original question via embedding similarity.

Low score means: answer is factual but off-topic or too vague.

📚 Context Recall

Did retrieval find the right chunks?

Compares retrieved chunks against ground-truth answers. What fraction of needed content was actually retrieved?

Low score means: retrieval is missing key information — fix chunking or retrieval. 
from ragas import evaluate
from ragas.metrics import faithfulness, answer_relevancy, context_recall
from datasets import Dataset

# ── Step 1: Build evaluation dataset ──
questions = [
    "How does token authentication work?",
    "What environment variables are required?",
    # … 8 more
]
ground_truths = [
    "Token auth uses JWT. The authenticate() function validates the token…",
    "Required env vars are: DATABASE_URL, SECRET_KEY, DEBUG…",
    # … 8 more (manually written)
]

# ── Step 2: Run pipeline on all questions ──
answers, contexts = [], []
for q in questions:
    result = rag_chain_with_sources.invoke(q)
    answers.append(result["answer"])
    contexts.append([d.page_content for d in result["source_documents"]])

# ── Step 3: Evaluate ──
dataset = Dataset.from_dict({
    "question":     questions,
    "answer":       answers,
    "contexts":     contexts,      # list of lists of strings
    "ground_truth": ground_truths  # needed for context_recall
})

result = evaluate(dataset, metrics=[
    faithfulness,      # no ground truth needed
    answer_relevancy,  # no ground truth needed
    context_recall,    # needs ground_truth
])
print(result.to_pandas())
Field Type Notes
question list[str] The test questions
answer list[str] Your RAG pipeline's output for each question
contexts list[list[str]] Chunks retrieved per question — each question gets its own list of chunk strings
ground_truth list[str] Manually written ideal answers. Required for context_recall and context_precision.
