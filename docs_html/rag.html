📚 RAG — Retrieval Augmented Generation

RAG grounds LLM responses in real documents. Instead of relying on the model's training data alone, RAG first retrieves relevant context from a knowledge base and then feeds it to the LLM for a more accurate, up-to-date answer.

✅ Learning Objectives After completing this page you will be able to:
  • Explain what RAG is and why it was invented
  • Describe LLM hallucinations, stale knowledge, and context-window constraints
  • Trace data through the full Ingestion → Retrieval → Generation pipeline
  • Understand embeddings and cosine similarity
  • Build a ChromaDB and PGVector-backed RAG chain from scratch
  • Compose the full RAG pipeline with LangChain LCEL and ChatGroq

What is RAG?

Retrieval-Augmented Generation (RAG) is an AI architecture pattern that enhances a language model's output by first retrieving relevant information from an external knowledge source and then injecting that information into the model's prompt before generation.

RAG has two stages: Retrieval (find the most relevant documents for a query) and Generation (have the LLM answer using those documents as context).

Plain-English Explanation

🧠 Analogy: The Open-Book Exam

Imagine you must answer questions about a 10,000-page technical manual. A closed-book LLM tries to answer from memory — it may be wrong, outdated, or confidently invent an answer that sounds plausible (hallucination).

RAG turns it into an open-book exam. Before answering, the system flips to the most relevant pages of the manual, reads them, then gives you a grounded, evidence-backed answer.

Without RAG, a user asking "What's our refund policy?" might get an invented, generic answer. With RAG, the system finds your actual policy document, reads it, and quotes the exact rules — attributable, accurate, and fresh.

Real-World Example

🏢 Customer Support Bot at a SaaS Company

Scenario: A software company has 500 help-centre articles, a changelog, and internal runbooks. They want an AI bot that answers user questions.

1
Without RAG: The LLM knows nothing about your product. It guesses: "You can reset your password in settings." — but your product calls it Account Security, not Settings. Users are confused. The bot hallucinates step numbers that don't exist.
2
With RAG:
  1. All 500 help articles are ingested — split into chunks, embedded, stored in ChromaDB.
  2. User asks: "How do I reset my 2FA?"
  3. The question is embedded; the 3 most similar chunks are retrieved instantly.
  4. Those chunks are injected into the LLM prompt alongside the question.
  5. The LLM reads the actual documentation and produces a precise, step-by-step answer.
3
Result: Answers are grounded in your real docs. When you update a policy, you just re-ingest the article — no re-training needed.

Why RAG Exists — LLM Limitations

RAG was invented to solve three fundamental problems with large language models:

Problem 1 — Hallucinations

LLMs are trained to produce plausible-sounding text — but plausible is not the same as true. When a model doesn't know an answer, it often generates a confident-sounding but fabricated response. This is called a hallucination.

❌ Hallucination Example

User: "Who wrote the internal API spec for project Orion at Acme Corp?"
LLM (without RAG): "The API spec for project Orion was authored by Sarah Chen, updated in Q3 2023." — Sarah Chen doesn't exist. The LLM invented a confident-sounding but completely false answer.

RAG prevents hallucinations by forcing the LLM to base its answer only on the retrieved context: if the answer isn't in the documents, the model is instructed to say so rather than invent one.

Problem 2 — Stale Knowledge (Training Cutoff)

Every LLM has a knowledge cutoff — a date after which it has no awareness of events, product updates, regulations, or new research. A model trained with data up to early 2024 knows nothing about what happened in late 2024 or 2025.

⚠️ Stale Knowledge Example

A model asked about the latest LangChain release notes, a new government regulation, or your company's Q4 2025 pricing will either hallucinate or refuse to answer — because none of that existed in its training data.

RAG solves stale knowledge cheaply: you just re-index your updated documents. No fine-tuning, no retraining, no GPU budget required.

Problem 3 — Context Window Limitations

Even the most capable LLMs have a context window limit — a maximum number of tokens (roughly words) they can process in a single call. Even with 128K-token windows, you cannot fit an entire knowledge base into every prompt. Attempting to do so is also slow and expensive.

📏 Context Window Math

500 help articles × avg 800 words = 400,000 words ≈ 500,000 tokens.
At $0.001 per 1K tokens × 1,000 queries/day = $500/day just in input tokens.
RAG retrieves only the relevant 3–5 chunks (~1,500 tokens) per query — a 300× cost reduction.

RAG selectively injects only the most relevant few chunks, keeping prompts short, fast, and cost-effective.

Summary: RAG vs No RAG

Problem Without RAG With RAG
Hallucinations LLM invents confident but false answers Answers grounded in retrieved evidence
Knowledge cutoff LLM only knows training-time data Retrieves current, fresh documents
Private / internal data LLM has no access whatsoever Your documents are indexed locally
Context window limits Can't fit large knowledge base in prompt Only injects the relevant top-K chunks
Update cost Fine-tuning = expensive GPU training Re-index new docs = cheap & instant

The RAG Pipeline

The RAG pipeline has two major phases: Ingestion (done once, offline) and Retrieval + Generation (done live, on every query). Here is the canonical end-to-end flow:

Pipeline Stage Breakdown

Each stage of the pipeline has a specific responsibility. Understanding each one will help you debug issues and tune performance.

1

Documents — Raw Knowledge Source

Your source material: PDF files, Markdown pages, HTML docs, database rows, API responses — anything with text. LangChain provides loaders for all common formats (PyPDFLoader, TextLoader, WebBaseLoader, etc.).

Key concern: Document quality directly determines answer quality. Garbage in = garbage out.

2

Chunking — Splitting Documents

Documents are too large to embed as a whole — the embedding model has a token limit, and large chunks lose precision. Chunking splits documents into overlapping segments. LangChain's RecursiveCharacterTextSplitter is the standard choice. 

from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,        # max characters per chunk
    chunk_overlap=50,      # characters shared between adjacent chunks
    separators=["\n\n", "\n", ".", " "],  # try these splits in order
)
chunks = splitter.split_documents(documents)

Key concern: Chunks too large → imprecise retrieval. Chunks too small → missing context. 300–800 tokens is typical.

3

Embeddings — Converting Text to Vectors

Each chunk of text is passed through an embedding model, which converts it into a dense numerical vector (e.g. 384 numbers). Semantically similar texts produce vectors that are mathematically close in that high-dimensional space.

Key concern: The embedding model must be consistent — you must use the same model to embed both documents and queries. Mixing models produces nonsensical similarity scores.

4

Vector Database — Indexed Storage

All chunk vectors are stored in a vector database (ChromaDB, PGVector, Pinecone, etc.) along with the original text as metadata. The database builds a specialised index (HNSW, IVFFlat) that enables approximate nearest-neighbour search over millions of vectors in milliseconds.

Key concern: For local dev use ChromaDB. For production with concurrent access, use PGVector or a managed service.

5

Retriever — Finding Relevant Chunks

At query time, the user's question is embedded using the same model. The retriever performs a similarity search against all stored chunk vectors, returning the top-K most similar chunks. LangChain's vectorstore.as_retriever(search_kwargs={"k": 3}) creates a retriever in one line.

Key concern: Choosing k is a trade-off: larger k = more context but longer prompt and higher cost.

6

LLM Prompt — Injecting Context

The retrieved chunks are concatenated into a context string and injected into the LLM prompt template alongside the original question. The system message explicitly instructs the model to answer only from the provided context, preventing hallucinations. 

rag_prompt = ChatPromptTemplate.from_messages([
    ("system", """Answer using ONLY the context below.
If the answer isn't there, say "I don't know."

Context:
{context}"""),
    ("human", "{question}"),
])
7

Answer — Grounded Generation

The LLM reads the injected context and generates a response that is grounded in your actual documents. Because the model is told to rely only on the context, the answer is traceable: you know exactly which chunks backed it up. This is the "Generation" in Retrieval-Augmented Generation.

What are Embeddings?

An embedding is a dense numerical vector that represents the semantic meaning of text. Texts with similar meanings produce vectors that are close together in high-dimensional space.

"The cat is sleeping on the sofa"
→
[0.12, -0.34, 0.89, ..., 0.22]
384 dimensions
"The kitten is resting on the couch"
→
[0.11, -0.33, 0.87, ..., 0.20]
Very similar! → cosine similarity ≈ 0.97

Notice the two cat/kitten sentences produce almost identical vectors — that's semantic similarity at work.

HuggingFace Embeddings

HuggingFaceEmbeddings runs an open-source embedding model locally, with no API cost. We use all-MiniLM-L6-v2 which produces 384-dimensional vectors. 

from langchain_huggingface import HuggingFaceEmbeddings

# Load the model (downloads ~90MB on first use)
embeddings = HuggingFaceEmbeddings(
    model_name="sentence-transformers/all-MiniLM-L6-v2",
    # Optional: run on GPU for speed
    model_kwargs={"device": "cpu"},   # "cuda" for GPU
    encode_kwargs={"normalize_embeddings": True},  # for cosine similarity
)

# Embed a single query
query_vector = embeddings.embed_query("What is AI?")
print(f"Vector dimensions: {len(query_vector)}")  # 384

# Embed multiple documents at once (more efficient)
docs = [
    "Artificial intelligence is transforming industries",
    "Machine learning is part of AI",
    "Dogs are loyal pets",
]
doc_vectors = embeddings.embed_documents(docs)
print(f"Shape: {len(doc_vectors)} docs × {len(doc_vectors[0])} dims")

HuggingFaceEmbeddings Parameters

Parameter Type Default Description
model_name str required HuggingFace model identifier, e.g. "sentence-transformers/all-MiniLM-L6-v2"
model_kwargs dict {} Passed to the model constructor. Use {"device": "cuda"} for GPU
encode_kwargs dict {} Passed to encode(). Set {"normalize_embeddings": True} to get unit vectors for cosine similarity
cache_folder str ~/.cache/huggingface Where to cache downloaded model weights

Cosine Similarity

Cosine similarity measures the angle between two vectors. A value of 1.0 means identical meaning, 0.0 means unrelated, and -1.0 means opposite. 

from langchain_huggingface import HuggingFaceEmbeddings
import numpy as np
from sklearn.metrics.pairwise import cosine_similarity

embeddings = HuggingFaceEmbeddings(model_name="sentence-transformers/all-MiniLM-L6-v2")

sentences = [
    "The cat is sleeping on the sofa",          # 0
    "A dog is playing in the park",              # 1
    "The kitten is resting on the couch",        # 2
    "Artificial intelligence is transforming technology",  # 3
    "Machine learning is a branch of AI",        # 4
]

vectors = np.array(embeddings.embed_documents(sentences))
similarity_matrix = cosine_similarity(vectors)

print("\nCosine Similarity Matrix:")
print("         cat    dog   kitten    AI      ML")
labels = ["cat  ", "dog  ", "kitten", "AI   ", "ML   "]
for i, label in enumerate(labels):
    row = "  ".join(f"{similarity_matrix[i][j]:.3f}" for j in range(len(sentences)))
    print(f"{label}: {row}")

# Expected: cat vs kitten ≈ 0.97 | AI vs ML ≈ 0.93 | cat vs AI ≈ 0.05

ChromaDB — Local Vector Store

ChromaDB stores embeddings locally on disk. No server required — perfect for development and single-machine RAG. 

from langchain_huggingface import HuggingFaceEmbeddings
from langchain_chroma import Chroma
from langchain_core.documents import Document

# 1. Set up embeddings
embeddings = HuggingFaceEmbeddings(model_name="sentence-transformers/all-MiniLM-L6-v2")

# 2. Create documents (can have page_content + metadata)
docs = [
    Document(page_content="Cats are small domesticated animals",
             metadata={"category": "animals", "source": "wiki"}),
    Document(page_content="Artificial intelligence is transforming industries",
             metadata={"category": "tech", "source": "wiki"}),
    Document(page_content="Machine learning is part of AI",
             metadata={"category": "tech", "source": "wiki"}),
    Document(page_content="ChromaDB is popular for local RAG systems",
             metadata={"category": "databases", "source": "docs"}),
    Document(page_content="LangChain simplifies LLM applications",
             metadata={"category": "tech", "source": "docs"}),
]

# 3. Create or load the vector store (persists to disk)
vectorstore = Chroma.from_documents(
    documents=docs,
    embedding=embeddings,
    persist_directory="./chroma_db",  # saved here
    collection_name="my_docs",        # group name inside ChromaDB
)

# 4. Similarity search
results = vectorstore.similarity_search("What is AI?", k=3)
for doc in results:
    print(f"• {doc.page_content}")
    print(f"  source: {doc.metadata.get('source')}")

Chroma.from_documents() Parameters

Parameter Type Description
documents list[Document] The documents to embed and store
embedding Embeddings The embedding model to use
persist_directory str Path where ChromaDB saves its data files
collection_name str Named collection within the ChromaDB database
ids list[str] Optional: explicit document IDs

Loading an existing ChromaDB

# Load a previously created ChromaDB (no re-embedding needed)
vectorstore = Chroma(
    persist_directory="./chroma_db",
    embedding_function=embeddings,
    collection_name="my_docs",
)

# similarity_search_with_score returns (doc, score) pairs
results_with_scores = vectorstore.similarity_search_with_score("What is AI?", k=3)
for doc, score in results_with_scores:
    print(f"Score: {score:.4f} | {doc.page_content}")

PGVector — PostgreSQL Vector Store

PGVector stores embeddings in PostgreSQL. Great for production where you need the reliability of Postgres, multi-user access, and advanced filtering.

📋 Prerequisites: PostgreSQL with pgvector extension 
 # Using Docker
docker run -d \
  -e POSTGRES_USER=postgres \
  -e POSTGRES_PASSWORD=postgres \
  -e POSTGRES_DB=postgres \
  -p 5432:5432 \
  pgvector/pgvector:pg16

# Or install the extension in existing Postgres:
# CREATE EXTENSION vector;
from langchain_huggingface import HuggingFaceEmbeddings
from langchain_postgres import PGVector
from langchain_core.documents import Document

# Connection string (SQLAlchemy format)
connection = "postgresql+psycopg://postgres:postgres@localhost:5432/postgres"

embeddings = HuggingFaceEmbeddings(model_name="sentence-transformers/all-MiniLM-L6-v2")

docs = [
    Document(page_content="PostgreSQL is a relational database"),
    Document(page_content="Vector databases enable semantic search"),
    Document(page_content="LangChain simplifies LLM applications"),
    Document(page_content="Deep learning uses neural networks"),
    Document(page_content="Embeddings convert text to vectors"),
]

# Create and populate the vector store
vectorstore = PGVector.from_documents(
    documents=docs,
    embedding=embeddings,
    connection=connection,
    collection_name="documents",         # stored as a separate table
    use_jsonb=True,                       # metadata stored as JSONB
    pre_delete_collection=False,         # set True to wipe and re-create
)

# Query
results = vectorstore.similarity_search("What is AI?", k=3)
for r in results:
    print(r.page_content)

PGVector.from_documents() Parameters

Parameter Type Description
documents list[Document] Documents to embed and store
embedding Embeddings Embedding model to use
connection str SQLAlchemy connection string
collection_name str Table name in PostgreSQL for this collection
use_jsonb bool Store metadata as JSONB (enables metadata filtering)
pre_delete_collection bool Delete existing data before inserting (use carefully)

ChromaDB vs PGVector

Feature ChromaDB PGVector
Setup complexity Zero — just a folder Requires PostgreSQL
Persistence File-based (SQLite) Full PostgreSQL ACID
Scalability Good for prototypes (<1M docs) Production-scale
Multi-user access Single-process only Full concurrent access
Metadata filtering Basic filtering Full SQL WHERE clauses
Index type HNSW (in-memory) IVFFlat or HNSW
Best for Local development, prototypes Production, team environments

Retrieval Methods

# Method 1: Simple similarity search
results = vectorstore.similarity_search(
    query="What is machine learning?",
    k=3,  # return top 3 most similar documents
)

# Method 2: Similarity search with scores
results_with_scores = vectorstore.similarity_search_with_score(
    query="What is machine learning?",
    k=3,
)
for doc, score in results_with_scores:
    print(f"Score: {score:.4f} — {doc.page_content[:60]}")

# Method 3: MMR — Maximal Marginal Relevance (reduces redundancy)
results_mmr = vectorstore.max_marginal_relevance_search(
    query="What is AI?",
    k=3,               # number of results
    fetch_k=10,        # fetch more candidates then diversify
    lambda_mult=0.5,   # 0 = max diversity, 1 = max relevance
)

# Method 4: Metadata filtering (PGVector / Chroma)
results_filtered = vectorstore.similarity_search(
    query="AI",
    k=3,
    filter={"category": "tech"},  # only return tech documents
)

Generation — Using Docs in LLM 

from langchain_core.prompts import ChatPromptTemplate
from langchain_groq import ChatGroq
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnablePassthrough

def format_docs(docs):
    """Combine retrieved documents into a single context string."""
    return "\n\n".join(doc.page_content for doc in docs)

# RAG prompt template
rag_prompt = ChatPromptTemplate.from_messages([
    ("system", """You are a helpful assistant. Answer the question using ONLY the provided context.
If the context doesn't contain the answer, say "I don't have that information."

Context:
{context}"""),
    ("human", "{question}"),
])

llm = ChatGroq(model="openai/gpt-oss-120b", temperature=0)
retriever = vectorstore.as_retriever(search_kwargs={"k": 3})

# Build the RAG chain
rag_chain = (
    {"context": retriever | format_docs, "question": RunnablePassthrough()}
    | rag_prompt
    | llm
    | StrOutputParser()
)

# Run it
answer = rag_chain.invoke("What is AI?")
print(answer)

Complete RAG Pipeline

"""Complete RAG pipeline: index documents, then query with LLM grounding."""
from langchain_huggingface import HuggingFaceEmbeddings
from langchain_chroma import Chroma
from langchain_core.documents import Document
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnablePassthrough
from langchain_groq import ChatGroq
from dotenv import load_dotenv

load_dotenv()

# ── 1. Load / create knowledge base ─────────────────────────────
embeddings = HuggingFaceEmbeddings(model_name="sentence-transformers/all-MiniLM-L6-v2")

docs = [
    Document(page_content="LangChain is a framework for building LLM applications"),
    Document(page_content="LangGraph is used to build stateful agents and multi-agent systems"),
    Document(page_content="FastMCP is a Python framework for building MCP servers quickly"),
    Document(page_content="ChromaDB stores embeddings locally and supports similarity search"),
    Document(page_content="Groq provides ultra-low-latency LLM inference via API"),
    Document(page_content="RAG stands for Retrieval Augmented Generation"),
]

vectorstore = Chroma.from_documents(docs, embeddings, persist_directory="./chroma_db")

# ── 2. Create retriever ──────────────────────────────────────────
retriever = vectorstore.as_retriever(search_kwargs={"k": 3})

# ── 3. Build RAG prompt ──────────────────────────────────────────
prompt = ChatPromptTemplate.from_messages([
    ("system", """Answer using ONLY the context below.
If the answer is not in the context, say "I don't know".

Context:
{context}"""),
    ("human", "{question}"),
])

llm = ChatGroq(model="openai/gpt-oss-120b", temperature=0)

def format_docs(docs):
    return "\n\n".join(d.page_content for d in docs)

# ── 4. Compose the full chain ────────────────────────────────────
rag_chain = (
    {"context": retriever | format_docs, "question": RunnablePassthrough()}
    | prompt
    | llm
    | StrOutputParser()
)

# ── 5. Query ─────────────────────────────────────────────────────
questions = [
    "What is FastMCP?",
    "How does RAG work?",
    "What is the difference between LangChain and LangGraph?",
]

for q in questions:
    print(f"\nQ: {q}")
    print(f"A: {rag_chain.invoke(q)}")

📕 Task 3 — Build the Full RAG Chain

Goal: Combine ChromaDB retrieval with ChatGroq to build a question-answering system. 
# Save as rag_full.py and run with: python rag_full.py

from langchain_huggingface import HuggingFaceEmbeddings
from langchain_chroma import Chroma
from langchain_core.documents import Document
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnablePassthrough
from langchain_groq import ChatGroq
from dotenv import load_dotenv

load_dotenv()

embeddings = HuggingFaceEmbeddings(model_name="sentence-transformers/all-MiniLM-L6-v2")
docs = [
    Document(page_content="The Eiffel Tower is in Paris, France"),
    Document(page_content="The Great Wall of China is over 13,000 miles long"),
    Document(page_content="Python was created by Guido van Rossum"),
    Document(page_content="LangChain makes it easy to build LLM applications"),
]
vs = Chroma.from_documents(docs, embeddings)
retriever = vs.as_retriever(search_kwargs={"k": 2})

prompt = ChatPromptTemplate.from_messages([
    ("system", "Answer based on context only:\n\n{context}"),
    ("human", "{question}"),
])

chain = (
    {"context": retriever | (lambda docs: "\n".join(d.page_content for d in docs)),
     "question": RunnablePassthrough()}
    | prompt | ChatGroq(model="openai/gpt-oss-120b", temperature=0) | StrOutputParser()
)

print(chain.invoke("Where is the Eiffel Tower?"))
print(chain.invoke("Who created Python?"))
